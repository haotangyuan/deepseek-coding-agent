# 高区分度任务 Prompt Profile A/B

> 执行日期：2026-07-16
> 产品提交：`c45bc8f0acbcbcbee368833b66c675179f824e25`
> Pi 研究基线：`dcfe36c79702ec240b146c45f167ab75ecddd205`
> Pi SDK：`@earendil-works/pi-coding-agent@0.80.7`
> 模型与档位：`deepseek-v4-flash` / `high`

## 1. 问题

首轮 Prompt A/B 的三个小任务在两档均为 9/9，无法区分质量。本轮改用跨模块探索、长日志定位和隐藏验证恢复，回答两个问题：`deepseek` Profile 是否在更接近日常 Coding Agent 的任务上提高完成率；其“窄搜索、避免重复工具”约束是否能降低工具错误、耗时或成本。

## 2. 方法

- 固定同一产品提交、Pi SDK、模型、thinking、fixture 和 evaluator，只改变 `--prompt-profile pi|deepseek`。
- 三个任务每档各运行 3 次，共 18 个逻辑样本；validation 每个样本最多两次 Provider 请求。
- 每个任务/档位单独设置 `$0.01` 观测上限，运行前已完成 dry-run。
- 只批准临时 fixture 内的 write/edit，拒绝 Bash；由工作区外测试确定评分。
- 不保存完整 reasoning、工具结果、Session JSONL 或密钥，只记录聚合指标。

## 3. 结果

|---|---|---:|---:|---:|---:|---:|---:|
| repair-cross-module | pi | 3/3 | 14705ms | $0.0028009 | 31 | 0 | 3 |
| repair-cross-module | deepseek | 3/3 | 14574ms | $0.0030356 | 31 | 0 | 3 |
| repair-long-log | pi | 3/3 | 8411ms | $0.0063728 | 12 | 1 | 3 |
| repair-long-log | deepseek | 3/3 | 7397ms | $0.0062076 | 10 | 0 | 3 |
| repair-validation | pi | 3/3 | 22796ms | $0.0038399 | 30 | 3 | 6 |
| repair-validation | deepseek | 3/3 | 28898ms | $0.0044131 | 28 | 2 | 6 |
| **合计** | **pi** | **9/9** | **15304ms/样本** | **$0.0130137** | **73** | **4** | **12** |
| **合计** | **deepseek** | **9/9** | **16957ms/样本** | **$0.0136563** | **69** | **2** | **12** |

两档 18 个逻辑样本共使用 24 次 Provider 请求，总成本 `$0.0266699944`，均低于各任务运行前设置的观测上限。

相对 `pi`，`deepseek` 的总体平均耗时高 10.8%、总成本高 4.9%，工具调用少 4 次、工具错误少 2 次。细分差异并不一致：

- 跨模块任务成功率与工具质量相同，DeepSeek 耗时低 0.9%，成本高 8.4%。
- 长日志任务中 DeepSeek 耗时低 12.1%、成本低 2.6%，工具调用从 12 降至 10，错误从 1 降至 0。
- validation 中两档均三次经历“首轮失败 → 摘要反馈 → 第二轮恢复”；DeepSeek 耗时高 26.8%、成本高 14.9%，虽然工具调用和错误略少，但不足以抵消效率回退。

两档均无 Provider 错误。缓存命中是请求前缀和服务端 best-effort 状态共同作用的结果，本轮没有人为清空或预热，不把自然缓存差异解释为 Prompt 因果。

## 4. 决策

1. 默认继续使用 `pi` Profile；18 个高区分度样本没有成功率提升，总体效率反而下降。
2. 保留 `deepseek` 显式入口。它对长日志表现出方向性收益，但样本不足以支持默认切换。
3. 不实现按任务自动 Profile 路由。隐藏路由会改变前缀、缓存和可重复性，当前只有一个任务出现局部收益。
4. 不针对 validation 的随机工具错误修改 Agent Loop 或做模糊参数修复；两档均 3/3 恢复，错误数量没有表现为完成阻断。
5. 下一阶段从 Prompt 调词转向只读 diagnostics/语义导航，先改善真实仓库定位能力，再用同一 suite 验证。

## 5. 结论边界

这组结论只适用于当前三个 fixture、Flash/high 和记录的提交。它证明现有短 Prompt 仍没有成为默认路径的依据，不证明窄搜索提示对更大真实日志无效，也不证明工具错误减少没有产品价值。
